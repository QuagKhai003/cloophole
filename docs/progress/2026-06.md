## 2026-06-23 — fix: GUI opened hidden — drop SW_HIDE (B10)
Branch `fix/gui-window-hidden-swhide`. User-reported: after the latest install,
`cloophole open` shows neither a window nor a terminal.

- **Root cause** — the B8 fix added `STARTUPINFO` `STARTF_USESHOWWINDOW | SW_HIDE` to
  suppress the console. `wShowWindow=SW_HIDE` is the child's `nCmdShow`, which Tk
  applies to its first top-level window → the GUI was created hidden. B8's source test
  only checked "process alive", not "window visible", so it slipped through.
- **Fix** — `CREATE_NO_WINDOW` alone: suppresses the console for the console-subsystem
  exe without hiding the GUI window. Dropped the SW_HIDE STARTUPINFO. Test updated to
  assert no STARTF_USESHOWWINDOW.
- **Verification limit** — only reproducible/confirmable on the frozen exe (source
  spawns windowless pythonw); per user's standing rule I do not launch the app here,
  so the user builds/installs and confirms. **29 tests.**
- **Process note** — three GUI-launch bugs in a row (B7 crash, B8 console, B10 hidden)
  all live in `runner._spawn` flag handling; none are caught by the offline suite
  because they're OS-window behaviors. Real fix verification = user on the exe.

## 2026-06-23 — fix: idle probe drained quota — poll OFF by default (B9)
Branch `fix/poll-off-by-default`. User-reported: "drained my usage while doing
nothing." Root cause: `poll_enabled` defaulted **on**, so the daemon sent a real
`claude -p` probe every 30 min while WATCHING — each costs quota, even when not
limited. A design flaw: it spends the resource the tool protects.

- **Mitigation** — `config.DEFAULTS["poll_enabled"]` → **False** (opt-in via `poll on`
  or the GUI checkbox). GUI checkbox now warns it spends usage. Test flipped
  (`test_poll_off_by_default`). DATA_MODEL + BUGS B9 logged.
- **Verified** — `last_fired`/`last_poll` were null in state, ruling out a runaway
  resume loop; the probe is the only quota path, hence the default flip.
- **Planned real fix** — replace polling with a Claude Code `StopFailure`/`rate_limit`
  hook: fires exactly when the limit hits, costs nothing, Golden-Rule clean (public
  hooks API, not internal reads). Hook `cwd` payload also fixes B6. Needs an ADR.
- **Tests:** 29 (unchanged count; one rewritten).

## 2026-06-22 — fix: blank console window + clipped GUI buttons (B8)
Branch `fix/gui-window-console-and-size`. After B7 the window appeared, but with a
blank black terminal behind it, and its bottom buttons were cut off (no scroll).

- **Console window** — `_spawn` passed `DETACHED_PROCESS | CREATE_NO_WINDOW` together.
  Win32 *ignores* CREATE_NO_WINDOW when DETACHED_PROCESS is set, so the console-
  subsystem exe got a visible blank console. (Latent until B7 let the GUI live.)
  **Fix:** CREATE_NO_WINDOW alone (no DETACHED) + STARTUPINFO SW_HIDE. The child still
  outlives the parent (separate process). +1 test asserting the flags.
- **Window size** — 440x430 clipped the 6 action buttons. **Fix:** taller default +
  clamp geometry/minsize to `winfo_reqheight()` after layout (DPI/font safe).
- **Note** — console fix only reproduces with the frozen console exe (source uses
  windowless pythonw); covered by unit test + Win32 docs, visually confirmed after CI
  rebuild. Window size verified live from source. **28 → 29** tests.
- **Docs** — BUGS B8 (resolved), STATUS, this entry.

## 2026-06-22 — fix: `cloophole open` showed no window (B7)
Branch `fix/detached-gui-stdio`. User-reported: fresh install + `cloophole open`
started the watcher but **no window appeared** (only a benign PyInstaller `_MEI`
temp-cleanup warning).

- **Diagnosis** — `gui.pid` was written but its process was dead, while the daemon
  lived. Foreground `_gui` ran fine (tkinter bundled OK); the detached spawn crashed.
  Reproduced: detached spawn **with** stdio→file/DEVNULL survives; **without** it dies.
- **Root cause** — `runner._spawn` used `DETACHED_PROCESS` (no console) but left stdio
  inherited → invalid stdout/stderr handles → GUI dies on Tk's first write, right
  after writing `gui.pid`. Daemon survived only because it doesn't write to stdout early.
- **Fix** — `_spawn` now passes `stdin/stdout/stderr=subprocess.DEVNULL`. Verified live
  from source (window survives 5s + clean stop). +1 regression test asserting the
  DEVNULL kwargs. **27 → 28** tests.
- **Docs** — BUGS B7 (resolved), STATUS (active task + "verify before trusting";
  corrected stale "pending merge `feat/gui-window`" — it was already merged).
- **Note** — released exe predates this; CI rebuilds on push to main, then reinstall.

## 2026-06-22 — RAM trims + footprint clarification
Branch `perf/ram-trim`. Tests **27**.

- **Reality check:** cloophole = 1 resident watcher (+1 window only while open); it does
  NOT spawn per `claude` terminal (just scans the process list). #terminals ≠ more RAM.
- **daemon `_trim_memory()`** — after startup: `gc.collect()` + `gc.freeze()` (move
  survivors out of GC tracking) + relaxed thresholds → less churn for the idle loop.
- **gui** — `gc.collect()` after building widgets, before idle.
- Honest floor: Python resident ~12-18 MB (source) / ~7 MB (frozen); the ~29 MB is the
  Tkinter window and only while open — close it to drop back to the watcher. Native
  (Rust) would shave only the small resident, not the window → not worth two toolchains.
