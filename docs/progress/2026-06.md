## 2026-06-23 — fix: `open` killed itself — kill_all spared no bootloader (B15)
Branch `fix/kill-all-bootloader-safe`. B14 wired `kill_all` into `cloophole open`, and
the window stopped appearing. Foreground `_gui` still worked, isolating it to the open
path.

- **Root cause** — PyInstaller onefile = bootloader (parent) + app (child), both
  `cloophole.exe`. `kill_all` excluded `os.getpid()` (the child) but `taskkill /T`'d the
  bootloader **parent**; the tree-kill cascaded back down to the child, so `open` killed
  itself before `launch_gui`.
- **Fix** — `winproc.list_procs(name)` returns `(pid, ppid)`; `kill_all` skips self AND
  its parent (the bootloader), still sweeping other instances (whose own bootloaders
  take their app children down via `/T`). Test rewritten to assert the bootloader is
  spared.
- **Tests:** 44 (one rewritten). Docs: BUGS B15, STATUS.
- Net: the B13/B14 flicker fixes + clean-restart now actually run, because `open` no
  longer self-terminates.

## 2026-06-23 — fix: onefile self-spawn ate the window + _MEI warning (B11)
Branch `fix/onefile-self-spawn-env`. After B10, `cloophole open` still showed no
window — but the decisive clue landed: a **python-parent** spawn of `cloophole.exe
_gui` with the *exact same* flags (CREATE_NO_WINDOW + DEVNULL) **opened the window**,
while the **frozen-parent** spawn (`cloophole open` → itself) did not.

- **Root cause** — PyInstaller onefile self-spawn. The frozen child inherited the
  parent's `_MEIPASS2`/`_PYI_*` env, so its bootloader attached to the PARENT's `_MEI`
  temp dir — which the parent deletes on exit (the long-standing "Failed to remove
  temporary directory _MEI..." warning). The child then couldn't load bundled tcl/tk,
  so no Tk window appeared. Flags were never the real issue.
- **Fix** — `runner._spawn` passes a child env stripped of `_MEI*`/`_PYI*` (new
  `_child_env()`), so the child extracts its own bundle like a clean launch. Fixes the
  no-window AND the `_MEI` warning in one go. +1 test. **29 → 30.**
- **Note** — the window itself was already proven good via foreground `_gui` (B7/B8/B10
  visible, no console, buttons fit); B11 was purely the launch/env path. User confirms
  on the rebuilt exe.
